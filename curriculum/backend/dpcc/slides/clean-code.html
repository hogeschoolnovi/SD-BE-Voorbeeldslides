Waarom Clean Code?

Niet alleen voor een hoger cijfer.

Waarom Clean Code?

  • Code schrijven bestaat vooral uit (andermans) code lezen
  • Een project bevindt zich 80% van de tijd in de onderhouds fase.
  • Je bent nooit de enige programmeur die aan een project werkt.
  • Verhoogt onderhoudbaarheid en uitbreidbaarheid van de applicatie.
  • Voorkomt een complete stilstand van nieuwe features.

Naamgeving

Hoe noem ik mijn packages, klassen, methodes en variabelen?

Naamgeving (1)

  • Klasse en packages zijn zelfstandige naamwoorden;
  • Klasse-, tabel- en packagenamen zijn altijd enkelvoud;
  • Methode-namen zijn werkwoorden (+ lijdend voorwerp)
  • Alle namen moeten niet ambigious zijn;
  • Namen moeten uitleggen of verklaren wat de inhoud is.

Naamgeving (2)

  • Gebruik de Java Code Conventions
  • https://www.oracle.com/java/technologies/javase/codeconventions-introduction.html
  • Blader het eens door.
  • Hoofdstuk 9 is onderdeel van eindtoets beoordeling.

Naamgeving (3)

  • Wees niet grappig
  • Gebruik geen generieke namen, zoals Manager, Processor, Data etc.
  • Gebruik Computer-Science terminologie
  • Voeg geen prefix toe aan klassenamen.
  • Gebruik geen (absoluut geen) Hongaarse notatie.

Nette Methodes

Meer dan een werkwoord + lijdend voorwerp

Regels

  • Houd je methodes klein!
  • Houd je methodes nog kleiner dan regel 1

Regels - SRP

Methodes moeten één ding doen. Dit moeten ze goed doen en ze moeten alleen dat doen.

Regels - Stepdown Rule

Zorg ervoor dat een methode die een andere methode aanroept boven de aangeroepte methode staat.

We zijn getraind om van boven naar beneden te lezen.

Regels - Aantal argumenten

  • Probeer het aantal argumenten zo klein mogelijk te houden.
  • Ideaal is nul.
  • Vier of meer is te veel
  • Gebruik builder of factory patterns.

Regels - Geen flag arguments

  • Maak geen methodes die iets doen op basis van een te ontvangen boolean - argument.
  • Maak liever twee methodes.

Command Query Seperation

  • Weer SRP!
  • Een methode moet een antwoord geven óf
  • Een methode moet data wijzigen
  • Niet allebei tegelijk!

CQRS - Voorbeeld (1)


                public boolean set(String attribute, String value) {
                    // ...
                }

                if(set("city","Utrecht")) {
                    //...
                }

CQRS - Voorbeeld (2)

Maak er twee methodes van.


            if(attributeExists("city")) {
                setAttribute("city", "Utrecht");
            }

Commentaar

Don't comment bad code. Rewrite it.

Commentaar - wanneer niet?

Gebruik methode- en variabelenamen om je code begrijpbaar te maken


        // Check to see if employee is eligible for full benefits
        if ((employee.flags & HOURLY_FLAG) && (employee.age > 65)) {
            // ...
        }
        // Of
        if (employee.isEligibleForFullBenefits()) {
            // ..
        }

Commentaar - Wanneer niet?

  • Overtollig commentaar
  • Onbedoeld misleidend commentaar
  • Bij sluit accolades plaatsen
  • Uitgecommentarieerde code
  • Journal-commentaar
  • Positie-markering

Commentaar - Wanneer dan wel?

  • Public API - JavaDoc
  • Legal comments wanneer erom gevraagd wordt.
  • JavaDoc intern (wanneer de opdrachtgever daarom vraagt)
  • Verduidelijking of verantwoording
  • Als waarschuwing.
  • TODO comments (Het verschilt per team of dit ook in master-branches mag zitten.)

Boy Scout Rule
