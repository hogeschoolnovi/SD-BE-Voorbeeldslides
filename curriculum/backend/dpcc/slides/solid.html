Single Responsibility Principle

"A class should be having one and only one responsibility."

Class A:

  • Open a database connection
  • Fetch data from database
  • Write the data in an external file

Klasse zou veranderd moeten worden bij een nieuwe database, het invoegen van Spring of het veranderen van de output.

Open/Closed Principle

"Classes should be open for extension but closed for modification."

Als je een klasse hebt gemaakt en iemand anders wil deze gebruiken of aanpassen, dan moet dat kunnen zonder dat de klasse wordt aangepast. De klasse zou moeten kunnen worden extend en de methodes zouden overriden moeten kunnen worden.

Liskov’s Substitution Principle

"Parent classes should be easily substituted with their child classes without blowing up the application."
Stel je voor je hebt een Animal class en die wordt extend door Dog en je wilt vervolgens een Cat toevoegen, dan mogen er geen methodes inzitten, dat als de parent class wordt aangeroepen dat de applicatie dan crasht.

Interface Segregation Principle

"Many client specific interfaces are better than one general interface."
Je wilt je interfaces zo klein mogelijk houden om te voorkomen dat je nutteloze methodes moet implementeren.

Dependency Inversion Principle

"Classes should depend on abstraction but not on concretion."

Dit is de reden dat je me altijd tegen List ziet aan programmeren en niet tegen ArrayList.

Dit zorgt ervoor dat de implementatie van Klasse A wordt 'verstopt' voor klasse B. Dit heeft als voordel dat als A verandert, dan hoeft B dat niet te weten of er rekening mee te houden.

Dit hangt samen met loose- en tight coupling
